## 1. 性能基础

### 常见性能问题

大多数 World 性能问题来自几个常见领域：

| 问题 | 影响 |
|------|------|
| 几何体过多 | CPU/GPU 负载 |
| 材质/绘制调用过多 | 渲染开销 |
| 大纹理或压缩不足 | 内存/加载 |
| 昂贵光照 | 实时计算 |
| 重型 Shader | GPU 负载 |

### 性能良好的表现

- 移动流畅
- 繁忙区域稳定
- 硬件适应性强
- 视觉噪音少

### 推荐检查点

1. 测试最繁忙的玩家区域：出生点、社交中心、活动空间、镜子、媒体区
2. 先减少最大的可见成本，再追逐小设置
3. 用真实 Avatar、平台目标、光照、镜子、世界系统重新测试

## 2. 预算规划

### 预算领域

| 领域 | 说明 |
|------|------|
| Mesh 复杂度 | 多边形数量 |
| 纹理分辨率 | 纹理大小 |
| 材质数量 | Material Count |
| 动态光使用 | Real-time Lights |
| Shader 成本 | Shader Complexity |
| 后处理 | Post-processing |
| 特效系统 | Mirrors/Video/Particles |

### 问自己

- 这是小型社交房间？
- 这是大型户外环境？
- 适合亲密聚会还是大型群体？
- 依赖实时效果还是烘焙视觉？

## 3. 材质管理

### 常见问题

创建许多几乎相同但无理由的材质。

### 最佳实践

- ✅ 尽可能复用材质
- ✅ 保持较小一致的材质集
- ❌ 避免大量微小重复

### 性能影响

- 更多材质 = 更多绘制调用
- 材质蔓延 = 项目更难管理
- 更难识别真正昂贵的部分

### 检查方法

如果两个对象看起来几乎相同，问自己：它们真的需要单独的材质吗？

## 4. Shader 使用

### 常见问题

Shader 是最大的隐藏性能成本之一。

### 最佳实践

| 环境工作 | 建议 |
|----------|------|
| 简化 Shader | ✅ 优先使用简单 Shader |
| 大数量对象 | ❌ 避免不必要重效果 |
| 透明效果 | ⚠️ 谨慎使用多层效果 |
| 动画材质 | ⚠️ 谨慎使用高级效果 |

### 问自己

- 这个效果对玩家体验重要吗？
- 需要应用到所有地方吗？
- 简单 Shader 能达到大部分效果吗？

### Quest/Android 建议

使用优化的移动端导向 Shader，如 VRChat 的 lightmapped mobile world shader。

## 5. 纹理管理

### 规则

| 纹理类型 | 建议 |
|----------|------|
| 大墙/地板 | 合理分辨率和压缩，避免过大纹理 |
| 小道具 | 玩家不仔细检查时积极降低 Max Size |
| UI 和标志 | 保持可读性后谨慎压缩 |
| 法线/遮罩图 | 也检查，可能和颜色纹理一样浪费 |
| 隐藏/未使用导入 | 发布前移除或缩小 |

### 关键点

- ✅ 仅在玩家实际注意到的地方使用高分辨率
- ✅ 对远距离或低重要性资产降低分辨率
- ✅ 质量可接受时压缩纹理
- ❌ 不要默认保持全分辨率

## 6. 几何体优化

### 检查项

- 在简单形式可工作的地方是否有非常密集的网格
- 是否有添加很少价值的装饰对象
- 是否有大量重叠或隐藏几何体
- 是否有太多做类似工作的独立对象

### 问自己

- 玩家会注意到这个细节吗？
- 这个几何体在正常游戏距离重要吗？
- 这个形状可以更简单吗？

## 7. 光照系统

### 烘焙 vs 动态

| 类型 | 适用场景 | 成本 |
|------|----------|------|
| **烘焙** | 静态区域、道具、墙壁、地板 | 低（离线计算） |
| **动态** | 需要改变的灯光、移动灯光、闪烁效果 | 高（实时计算） |

### 动态光常见问题

- 太多点光源在一个区域
- 光照范围过大
- 光照影响太多对象
- 不有意义改善场景的装饰灯

### Quest/Android 建议

- VRChat 推荐大量使用烘焙光照
- 使用 Light Probes 让 Avatar 和动态对象接收采样光照

## 8. 场景设计

### 可见性控制

**问题**: 一切都同时可见的世界通常更难优化

**解决方案**: 设计场景让可见性自然受控

| 方法 | 说明 |
|------|------|
| 房间和走廊 | 分割室内空间 |
| 角落/墙壁/高度变化 | 限制可见性 |
| 区域结构 | 不是每个对象从每个视角渲染 |

### 支持技术

- 遮挡剔除（Culling）
- 大场景更易管理

---

## 9. 动态系统

### 需要审查的系统

| 系统 | 成本 | 建议 |
|------|------|------|
| Reflection Probes | 中-高 | 谨慎使用 |
| Real-time Mirrors | 高 | 集中使用 |
| Post-processing | 中-高 | 适度使用 |
| 粒子效果 | 中-高 | 适度使用 |
| 脚本系统 | 取决于 | 检查更新频率 |

### 关键点

这些不是自动坏的。关键是在最重要的的地方使用它们，而不是无处不在。

## 10. 测试方法

### 真实条件测试

❌ **不要**: 在空的世界版本中单独测试性能，假设结果在真实使用中会保持

✅ **应该**: 在以下条件下测试：
- 主社交区域
- 有多个 Avatar 在场
- 重要效果启用
- 玩家花费最多时间的区域

### 迭代循环

```
1. 识别感觉慢的区域
2. 做一个有意义的改变
3. 再次测试
4. 比较结果
5. 重复
```

### 良好第一目标

- 大纹理
- 昂贵 Shader
- 密集装饰道具
- 太多活动灯光
- 问题热点区域

## 11. 烘焙光照

### 适用场景

| 场景 | 烘焙光照帮助因为 |
|------|------------------|
| 静态房间、道具、墙、地板 | 光照可以存储在光照贴图中而非每帧重新计算 |
| Android/Quest 世界支持 | 烘焙光照通常比动态光照便宜得多 |
| 有很多 Avatar 的社交空间 | 运行时预算可以为玩家、镜子、音频、交互系统节省 |
| 风格化世界 | 烘焙光照可以锁定清晰的艺术方向，无需重实时效果 |
| 性能清理 | 减少对重叠动态灯和昂贵实时阴影的需求 |

### 烘焙光照检查清单

| 步骤 | 检查项 |
|------|--------|
| 静态几何体 | 仅将稳定环境网格标记为贡献烘焙光照 |
| 光照模式 | 固定环境光照使用烘焙灯 |
| Light Probes | 在可步行和 Avatar 可见区域放置探针 |
| Reflection Probes | 在闪亮材质、镜子或反射道具需要的地方添加覆盖 |
| 光照贴图分辨率 | 从适度开始，仅在结果需要更清晰阴影或细节时增加 |
| 测试构建 | 在 VRChat 中检查，不仅在 Unity Scene 视图中 |

### 常见烘焙光照问题

| 问题 | 解决方案 |
|------|----------|
| Unity 中好但 VRChat 中太暗/亮 | 检查曝光、天空盒、环境光照、反射探针、后处理、光照数据是否烘焙保存 |
| Avatar 与房间光照断开 | 在玩家高度空间周围添加或改进 Light Probes |
| 某些对象不接收烘焙光照 | 检查对象是否静态、渲染器是否能贡献全局光照、是否有可用光照贴图 UV、烘焙是否完成 |
| 烘焙时间长或数据过大 | 降低光照贴图分辨率、分成更小区域测试、降低不必要细节、避免使每个小道具成为高分辨率烘焙光照通道的一部分 |

## 常见初学者性能错误

1. **出生区太重** - 出生点是每个玩家首先看到世界的地方
2. **每个导入资产保持全质量** - 资产包通常包括比你需要的重得多的纹理、Shader 和网格
3. **过度使用视觉特效** - Bloom、透明、动画材质、粒子、镜子、花哨 Shader 会快速累积
4. **建造巨大的空旷空间** - 大空间通常需要更多可见几何体，可能更难优化
5. **延迟优化到结束** - 如果等太久，性能问题变得更难修复
